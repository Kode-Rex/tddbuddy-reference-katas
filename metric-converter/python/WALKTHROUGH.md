# Metric Converter вЂ” Python Walkthrough

This is an **algorithmic kata** with a small closed domain: one `StrEnum` вЂ” `Unit(KILOMETERS, MILES, CELSIUS, FAHRENHEIT, KILOGRAMS, POUNDS, LITERS, US_GALLONS, UK_GALLONS)` вЂ” and a pure function `convert(value, from_unit, to_unit) -> float`. There are no builders, no value types beyond the enum, and no collaborators вЂ” the enum *is* the domain. The reference lands as a single commit: `src/metric_converter/metric_converter.py` defines the enum and `convert`, which uses a structural `match` on the `(from_unit, to_unit)` tuple to dispatch to one of the five supported conversions вЂ” `KILOMETERSв†’MILES`, `CELSIUSв†’FAHRENHEIT`, `KILOGRAMSв†’POUNDS`, `LITERSв†’US_GALLONS`, `LITERSв†’UK_GALLONS`. The wildcard arm raises `ValueError` for unsupported pairs. The package `__init__.py` re-exports the two public names so tests import them as `from metric_converter import Unit, convert`. Constants are inline; the kata spec names them by value (`0.621371`, `0.45359237`, `3.785411784`, `4.54609`) and extracting named constants would restate the literal twice without adding meaning. `tests/test_metric_converter.py` has one function per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); approximate assertions use `pytest.approx`.

**`StrEnum` over strings вЂ” deliberate.** F1 normally keeps inputs as primitives, but the `from`/`to` pair is a closed set with business-named members. `StrEnum` gives a closed type whose members are readable in repr and stable across `==` and `match`. The C# and TypeScript siblings use their own casing conventions (`UsGallons`, `usGallons`); Python uses screaming snake case for enum members with lower-snake string values (`US_GALLONS = "us_gallons"`), idiomatic in each language while the vocabulary is shared.
