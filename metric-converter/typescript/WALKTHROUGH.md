# Metric Converter вЂ” TypeScript Walkthrough

This is an **algorithmic kata** with a small closed domain: one string-literal union вЂ” `type Unit = 'kilometers' | 'miles' | 'celsius' | 'fahrenheit' | 'kilograms' | 'pounds' | 'liters' | 'usGallons' | 'ukGallons'` вЂ” and a pure function `convert(value: number, from: Unit, to: Unit): number`. There are no builders, no value types beyond the union, and no collaborators вЂ” the union *is* the domain. The reference lands as a single commit: `src/metricConverter.ts` declares the union and `convert`, which composes the `(from, to)` pair into a switchable key and dispatches to one of the five supported conversions вЂ” `kilometersв†’miles`, `celsiusв†’fahrenheit`, `kilogramsв†’pounds`, `litersв†’usGallons`, `litersв†’ukGallons`. Unsupported pairs (e.g. `kilometersв†’fahrenheit`) throw `Error`. Constants are inline; the kata spec names them by value (`0.621371`, `0.45359237`, `3.785411784`, `4.54609`) and extracting them as named constants would restate the literal twice without adding meaning. `tests/metricConverter.test.ts` has one `it()` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); approximate assertions use Vitest's `toBeCloseTo` with a precision that catches formula errors while absorbing binary-float rounding.

**Typed union over strings вЂ” deliberate.** F1 normally keeps inputs as primitives, but here the `from`/`to` pair is a closed set with business-named members. A bare `string` would invite `"km"` vs `"kilometers"` drift at every call site; the union gives TypeScript exhaustiveness without a runtime class. Camel-case members (`usGallons`, `ukGallons`) match the TS idiom; the C# and Python siblings use their own casing conventions вЂ” same vocabulary, each language spells it idiomatically.
