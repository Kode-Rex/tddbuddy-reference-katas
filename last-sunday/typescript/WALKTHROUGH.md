# Last Sunday — TypeScript Walkthrough

This is an **algorithmic kata**: inputs are two `number`s (`year` and `month`, the latter 1-indexed per human convention) and the output is a standard-library `Date` — so there are no domain builders, no value types, and no collaborators to introduce. The reference lands as a single commit: `src/lastSunday.ts` exports `find(year, month): Date` which constructs the last day of the month as `new Date(Date.UTC(year, month, 0))` — passing `month` unadjusted with `day = 0` rolls back into the previous month, which is the 1-indexed month the caller asked about — then walks back `(getUTCDay() + 7) % 7` days to land on Sunday. `tests/lastSunday.test.ts` has one `it()` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); each test name reads as a sentence from that spec.

**UTC Date discipline — deliberate.** JavaScript's `Date` is a single UTC instant rendered in the host's local time zone; construct via `Date.UTC(...)` and read back via `getUTCDay()` / `getUTCDate()` so the returned date's *calendar components* always match the intended year/month/day regardless of where the code runs. The test-side `utc(y, m, d)` helper mirrors this discipline: tests assert on UTC-constructed Dates, not locally-constructed ones, so `toEqual` compares two instants at UTC midnight rather than accidentally drifting by a day in non-zero-offset zones.

**Inline literals — deliberate.** The `7` stays inline rather than hiding behind `DAYS_IN_WEEK = 7`. In F1 katas the function body fits on one screen and the literals *are* the rule. F3 and larger modes prefer named constants; F1 deliberately doesn't.
