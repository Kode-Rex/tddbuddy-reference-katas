# Anagram Detector — Python Walkthrough

This is an **algorithmic kata**: inputs are `str` / `Iterable[str]`, outputs are `bool` and ordered `list[str]` / `list[list[str]]` — so there are no domain builders, no value types, and no collaborators to introduce. The reference lands as a single commit. `src/anagram_detector/anagram_detector.py` defines three pure functions: `are_anagrams(a, b)`, `find_anagrams(subject, candidates)`, and `group_anagrams(words)`. All three share a private `_key` helper that lowercases, keeps only `str.isalpha` characters, and sorts — the sorted-letters "anagram key" is the algorithmic insight. `are_anagrams` short-circuits the not-itself rule by comparing raw inputs with `==` before keying, and rejects empty-letter inputs so `("", "")` and `("!", "?")` come back `False`. `group_anagrams` preserves first-occurrence order by pairing a dict with an ordered `order` list of keys (Python dicts preserve insertion order too, but the explicit list makes the ordering intent obvious to readers). The package `__init__.py` re-exports all three so tests import them as `from anagram_detector import are_anagrams, find_anagrams, group_anagrams`. `tests/test_anagram_detector.py` has one function per scenario in [`../SCENARIOS.md`](../SCENARIOS.md).

**Inline literals — deliberate.** Scenario literals like `"listen"`, `"silent"`, and `["eat", "tea", "ate"]` sit inline in the tests rather than behind named builders. In F1 katas the inputs *are* the domain — extracting `CANONICAL_ANAGRAM = "listen"` would write "listen" twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
