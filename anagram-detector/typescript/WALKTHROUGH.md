# Anagram Detector — TypeScript Walkthrough

This is an **algorithmic kata**: inputs are `string` / `readonly string[]`, outputs are `boolean` and ordered string arrays — so there are no domain builders, no value types, and no collaborators to introduce. The reference lands as a single commit. `src/anagramDetector.ts` exports three pure functions: `areAnagrams(a, b)` for pair detection, `findAnagrams(subject, candidates)` to filter a list, and `groupAnagrams(words)` to partition by anagram set. All three share a private `key` helper that lowercases, drops non-letters (`c >= 'a' && c <= 'z'`), and sorts — the sorted-letters "anagram key" is the algorithmic insight. `areAnagrams` short-circuits the not-itself rule by comparing raw inputs with `===` before keying, and rejects empty-letter inputs so `('', '')` and `('!', '?')` come back `false`. `groupAnagrams` preserves first-occurrence order using a `Map` alongside an ordered `order[]` array of keys. `tests/anagramDetector.test.ts` has one `it()` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md), grouped by surface (`areAnagrams`, `findAnagrams`, `groupAnagrams`) to match the exported shape rather than any namespace.

**Inline literals — deliberate.** Scenario literals like `'listen'`, `'silent'`, and `['eat', 'tea', 'ate']` sit inline in the tests rather than behind named builders. In F1 katas the inputs *are* the domain — extracting `const CANONICAL_ANAGRAM = 'listen'` would write "listen" twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
