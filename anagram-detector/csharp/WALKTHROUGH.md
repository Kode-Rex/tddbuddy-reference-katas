# Anagram Detector — C# Walkthrough

This is an **algorithmic kata**: inputs are `string` / `IEnumerable<string>`, outputs are `bool` and ordered string collections — so there are no domain builders, no value types, and no collaborators to introduce. The reference lands as a single commit. `src/AnagramDetector/AnagramDetector.cs` is a static class exposing three pure functions: `AreAnagrams(a, b)` for pair detection, `FindAnagrams(subject, candidates)` to filter a list, and `GroupAnagrams(words)` to partition by anagram set. All three share a private `Key` helper that lowercases, drops non-letters, and sorts — the sorted-letters "anagram key" is the algorithmic insight. `AreAnagrams` short-circuits the not-itself rule by comparing the raw inputs with `==` before keying, and also rejects empty-letter inputs so `("", "")` and `("!", "?")` come back `false`. `GroupAnagrams` preserves first-occurrence order for both groups and members by tracking insertion order alongside the key dictionary. `tests/AnagramDetector.Tests/AnagramDetectorTests.cs` has one `[Fact]` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); each test name reads as a sentence from that spec.

**Inline literals — deliberate.** Scenario literals like `"listen"`, `"silent"`, and `new[] { "eat", "tea", "ate" }` sit inline in the tests rather than behind named builders. In F1 katas the inputs *are* the domain — extracting `const string CanonicalAnagram = "listen"` would write "listen" twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
