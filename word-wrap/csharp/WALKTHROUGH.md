# Word Wrap — C# Walkthrough

This is an **algorithmic string-transform kata**: the input is a string plus an integer width, the output is a `\n`-joined string, and there are no aggregates, value types, or collaborators — the inputs and outputs *are* the domain. The reference lands as a single commit: `src/WordWrap/WordWrap.cs` exposes `WordWrap.Wrap(string text, int width)` returning `string`. A greedy line-packer accumulates words until the next word would push the line past `width`, opens a new line when it does, and splits any oversize word hard at the `width`-th character before placing its remainder. The lines are joined with `string.Join('\n', lines)` — matching the classic Kent Beck kata shape that returns a single string rather than a list. `tests/WordWrap.Tests/WordWrapTests.cs` has one `[Fact]` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); each test name reads as a sentence from that spec.

**Tokenization via `Split((char[]?)null, StringSplitOptions.RemoveEmptyEntries)`.** Passing `null` for the separator array is the .NET idiom for "split on any Unicode whitespace"; `RemoveEmptyEntries` drops empty tokens produced by runs of consecutive whitespace. This handles scenarios 1, 2, and 11 (empty, whitespace-only, and multi-space inputs) with a single API call and no special-casing. Collapsing whitespace is the spec's chosen behavior for the "multiple spaces" edge case — the classic Kent-Beck `rfind`-space approach can preserve gaps, but tokenization makes the rule "the input is a sequence of words" explicit.

**Oversize words split hard, then fall through to the greedy packer.** Inside the `foreach (var original in words)` loop, a `while (word.Length > width)` flushes full-width chunks as their own lines and leaves a remainder shorter than `width`. That remainder then enters the same branch as any other word — placed on a fresh line if one is active, or joined to the current line when it still fits. This is how scenario 10 (`"abcdefghij kl"`, width 5) produces `"abcde\nfghij\nkl"`: `"abcdefghij"` flushes `"abcde"` and leaves `"fghij"` (length 5), which starts the current line; `"kl"` does not fit (`5 + 1 + 2 > 5`) so it opens the final line.

**Inline literals — deliberate.** The whitespace set is delegated to the framework via `null` separators, the line separator is `'\n'` inline in `string.Join('\n', lines)`, and the intra-line joiner is the literal `' '` in `line.Append(' ').Append(word)`. Naming `NEWLINE = '\n'` or `SPACE = ' '` would substitute constant names for characters that *are* the rule. F1 katas treat the literal rules as the rule itself.
