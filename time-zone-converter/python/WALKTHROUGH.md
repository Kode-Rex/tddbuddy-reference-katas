# Time Zone Converter — Python Walkthrough

This is an **algorithmic kata** scoped to **fixed UTC offsets**: inputs are a naive `datetime`, a source `timedelta` offset, and a target `timedelta` offset; the output is a naive `datetime` at the target offset. There are no domain builders, no value types, and no collaborators — `datetime` and `timedelta` from the standard library already express "a wall-clock date-time" and "a signed duration from UTC" cleanly. The reference lands as a single commit: `src/time_zone_converter/time_zone_converter.py` defines `convert(local, from_offset, to_offset) -> datetime` whose body is one expression — `local - from_offset + to_offset` — re-exported through the package `__init__.py` so tests import it as `from time_zone_converter import convert`. `tests/test_time_zone_converter.py` has one function per scenario in [`../SCENARIOS.md`](../SCENARIOS.md).

**Scope — deliberate.** The kata spec's hint opens with "Start by implementing basic time zone conversions using fixed offsets, then add support for daylight saving time." This reference stays in step one. IANA zones (`zoneinfo.ZoneInfo("America/New_York")`), DST rules, and ambiguous/skipped local times around transitions are out of scope — adding them swaps the kata from F1-algorithmic to F3-domain-modeling.

**Naive datetimes — deliberate.** The function takes and returns naive `datetime` objects (`tzinfo=None`) because the caller already carries the offset externally as a separate `timedelta`. Attaching a `timezone(from_offset)` tzinfo to the input and calling `.astimezone(timezone(to_offset))` would work equivalently, but that route adds object construction and assumes the caller thinks of "local time plus offset" as one zoned thing rather than two independent inputs. The kata's surface is the latter.

**Inline arithmetic — deliberate.** `local - from_offset + to_offset` stays inline rather than routing through `normalise_to_utc` and `project_to_target` helpers. In F1 katas the function body fits on one line and the expression *is* the rule. `datetime` subtraction of a `timedelta` and addition of a `timedelta` carry across minute / hour / day / month / year boundaries using the standard library's calendar arithmetic, so the twelve scenarios (including Feb 29 → Mar 1 and Jan 1 → Dec 31 of the previous year) all fall out of the same one-line implementation without any explicit boundary handling.
