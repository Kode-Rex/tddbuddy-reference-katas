# Time Zone Converter — TypeScript Walkthrough

This is an **algorithmic kata** scoped to **fixed UTC offsets**: inputs are a naive `LocalDateTime` (a plain object of `year/month/day/hour/minute/second`) plus a source and target `Offset` (`{ hours, minutes }`); the output is a `LocalDateTime` at the target offset. There are no domain builders, no value types beyond these two shape-only interfaces, and no collaborators. The reference lands as a single commit: `src/timeZoneConverter.ts` exports `convert(local, fromOffset, toOffset)` which normalises the input to a UTC millisecond instant via `Date.UTC(...)`, shifts by `(toOffsetMinutes - fromOffsetMinutes) * 60_000`, then reads the target calendar components off a new `Date` via the `getUTC*` family. `tests/timeZoneConverter.test.ts` has one `it()` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); `local(...)` and `offset(...)` helpers keep each scenario a one-line readable call.

**Scope — deliberate.** The kata spec's hint opens with "Start by implementing basic time zone conversions using fixed offsets, then add support for daylight saving time." This reference stays in step one. IANA zones (`Intl.DateTimeFormat` with `timeZone: 'America/New_York'` or Luxon), DST rules, and ambiguous/skipped local times around transitions are out of scope — adding them swaps the kata from F1-algorithmic to F3-domain-modeling.

**UTC discipline — deliberate.** JavaScript's `Date` is a single UTC instant rendered in the host's local time zone by default. All arithmetic here uses `Date.UTC(...)` to construct and `getUTC*` to read back, so the returned calendar components are stable regardless of where the code runs. `noUncheckedIndexedAccess` in `tsconfig.json` is inherited from the sibling F1 katas — it's not load-bearing here because no array indexing occurs, but keeping strict settings consistent across the repo is the simpler rule.

**Offset sign convention — deliberate.** An offset of `-05:30` is passed as `{ hours: -5, minutes: -30 }` rather than `{ hours: -5, minutes: 30 }`. `totalMinutes` is then a plain `hours * 60 + minutes` rather than a sign-aware hack. The kata spec doesn't specify a wire format for offsets, so the one with less ceremony in the conversion function wins.

**Inline literals — deliberate.** `MS_PER_MINUTE = 60_000` is named because `60_000` in isolation could be "sixty thousand of what?" and the name pays for itself at the one call site. The `month - 1` / `getUTCMonth() + 1` adjustments stay inline; they *are* the rule (JavaScript months are 0-indexed, the public API is 1-indexed) and naming `JS_MONTH_OFFSET` would restate "one" twice.
