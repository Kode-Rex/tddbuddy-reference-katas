# Time Zone Converter — C# Walkthrough

This is an **algorithmic kata** scoped to **fixed UTC offsets**: inputs are a naive `DateTime`, a source `TimeSpan` offset, and a target `TimeSpan` offset; the output is a naive `DateTime` at the target offset. There are no domain builders, no value types, and no collaborators — `TimeSpan` and `DateTime` from the BCL already express "a signed duration from UTC" and "a wall-clock date-time" cleanly, so introducing an `Offset` wrapper would be ceremony without payoff. The reference lands as a single commit: `src/TimeZoneConverter/TimeZoneConverter.cs` exposes `public static DateTime Convert(DateTime local, TimeSpan fromOffset, TimeSpan toOffset)` whose body is one expression — `local - fromOffset + toOffset` — and `tests/TimeZoneConverter.Tests/TimeZoneConverterTests.cs` has one `[Fact]` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md).

**Scope — deliberate.** The kata spec's hint opens with "Start by implementing basic time zone conversions using fixed offsets, then add support for daylight saving time." This reference stays in step one. IANA zones (`TimeZoneInfo.FindSystemTimeZoneById("America/New_York")`), DST rules, and ambiguous/skipped local times around transitions are out of scope — adding them swaps the kata from F1-algorithmic to F3-domain-modeling (an `IClock`, an `IZoneRegistry`, a disambiguation policy for the repeated hour in fall-back). The scenarios exercise the fixed-offset arithmetic thoroughly: half-hour (`+05:30`) and quarter-hour (`+05:45`) offsets, day / month / year boundary roll-overs in both directions, a leap-day crossing, and a twenty-four-hour swing across the international date line.

**Inline arithmetic — deliberate.** `local - fromOffset + toOffset` stays inline rather than routing through a helper called `NormaliseToUtcThenProject`. In F1 katas the function body fits on one screen and the expression *is* the rule; naming the intermediate UTC instant would write "subtract source, add target" in prose above code that already reads that way. `DateTime` subtraction of a `TimeSpan` and addition of a `TimeSpan` carry across minute / hour / day / month / year boundaries using the BCL's calendar arithmetic, so the twelve scenarios (including Feb 29 → Mar 1 and Jan 1 → Dec 31 of the previous year) all fall out of the same one-line implementation without any explicit boundary handling.
