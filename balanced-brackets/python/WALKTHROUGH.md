# Balanced Brackets — Python Walkthrough

This is an **algorithmic kata**: the input is a `str`, the output is a `bool` — so there are no domain builders, no value types, and no collaborators to introduce. The reference lands as a single commit. `src/balanced_brackets/balanced_brackets.py` defines one pure function: `is_balanced(input)`. The algorithm is a single-pass depth counter — increment on `[`, decrement on `]`, fail fast when the counter goes negative, and require the final counter to be zero. A list used as a stack would work too, but because the only bracket type is `[]` the stack never carries information the counter can't — an `int` is the honest representation. The package `__init__.py` re-exports `is_balanced` so tests import it as `from balanced_brackets import is_balanced`. `tests/test_balanced_brackets.py` has one function per scenario in [`../SCENARIOS.md`](../SCENARIOS.md).

**Inline literals — deliberate.** Scenario literals like `"[[]]"` and `"[][]]["` sit inline in the tests rather than behind named constants. In F1 katas the inputs *are* the domain — extracting `DEEPLY_NESTED = "[[[][]]]"` would write the string twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
