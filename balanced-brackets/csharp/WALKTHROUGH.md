# Balanced Brackets — C# Walkthrough

This is an **algorithmic kata**: the input is a `string`, the output is a `bool` — so there are no domain builders, no value types, and no collaborators to introduce. The reference lands as a single commit. `src/BalancedBrackets/BalancedBrackets.cs` is a static class exposing one pure function: `IsBalanced(input)`. The algorithm is a single-pass depth counter — increment on `[`, decrement on `]`, fail fast when the counter goes negative (a `]` with no open `[` to its left), and require the final counter to be zero. A `Stack<char>` would work too, but because the only bracket type is `[]` the stack never carries information the counter can't — an `int` is the honest representation. `tests/BalancedBrackets.Tests/BalancedBracketsTests.cs` has one `[Fact]` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); each test name reads as a sentence from that spec.

**Inline literals — deliberate.** Scenario literals like `"[[]]"` and `"[][]]["` sit inline in the tests rather than behind named constants. In F1 katas the inputs *are* the domain — extracting `const string DeeplyNested = "[[[][]]]"` would write the string twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
