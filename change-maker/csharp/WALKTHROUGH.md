# Change Maker — C# Walkthrough

This is an **algorithmic kata**: the input is an `int` amount and an `IReadOnlyList<int>` of denominations, the output is an `IReadOnlyList<int>` of coins — so there are no domain builders, no value types, and no collaborators to introduce. The inputs and outputs *are* the domain. The reference lands as a single commit: `src/ChangeMaker/ChangeMaker.cs` exposes a static `MakeChange(int amount, IReadOnlyList<int> denominations)` that walks denominations largest-to-smallest, taking each coin while it fits in the remaining amount. `tests/ChangeMaker.Tests/ChangeMakerTests.cs` has one `[Fact]` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); three currency-specific `readonly` arrays (`UsCoins`, `UkCoins`, `NorwayCoins`) keep the test names in business language — "forty-one cents," "eighty-eight pence," "thirty-seven øre" — rather than anonymous int literals.

**Greedy is correct here, not generally.** The algorithm is a textbook greedy: largest coin that fits, repeat. For the three canonical currencies in this kata (US 25/10/5/1, UK 50/20/10/5/2/1, Norway 20/10/5/1) greedy yields the minimum-coin answer. For a non-canonical set like `{1, 3, 4}` making `6`, greedy picks `4+1+1` while the optimum is `3+3` — solving the general minimum-coin problem requires dynamic programming. This reference deliberately stops at the canonical-currencies scope of the TDD Buddy prompt.

**Inline literals — deliberate.** The currency coin arrays are kept as inline `readonly int[]` fields in the test class rather than extracted into a `Currencies` helper. In F1 katas the test fixture fits on one screen and the coin lists *are* the test data — wrapping them in a helper type would hide the business vocabulary the test names already carry. F3 and larger modes prefer typed value objects; F1 deliberately doesn't.
