# Change Maker — Python Walkthrough

This is an **algorithmic kata**: the input is an `int` amount and a `Sequence[int]` of denominations, the output is a `list[int]` of coins — so there are no domain builders, no value types, and no collaborators to introduce. The inputs and outputs *are* the domain. The reference lands as a single commit: `src/change_maker/change_maker.py` defines `make_change(amount, denominations)` that walks denominations largest-to-smallest, taking each coin while it fits in the remaining amount. The package `__init__.py` re-exports `make_change` so tests import it as `from change_maker import make_change`. `tests/test_change_maker.py` has one function per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); three module-level tuples (`US_COINS`, `UK_COINS`, `NORWAY_COINS`) keep test names in business vocabulary.

**Greedy is correct here, not generally.** The algorithm is a textbook greedy: largest coin that fits, repeat. For the three canonical currencies in this kata greedy yields the minimum-coin answer. For a non-canonical set like `(1, 3, 4)` making `6`, greedy picks `[4, 1, 1]` while the optimum is `[3, 3]` — solving the general minimum-coin problem requires dynamic programming. This reference deliberately stops at the canonical-currencies scope of the TDD Buddy prompt.

**Inline literals — deliberate.** The coin tuples are module-level constants in the test file rather than extracted into a `currencies` module. In F1 katas the test fixture fits on one screen and the coin lists *are* the test data — wrapping them would hide the vocabulary the test names already carry. F3 and larger modes prefer typed value objects; F1 deliberately doesn't.
