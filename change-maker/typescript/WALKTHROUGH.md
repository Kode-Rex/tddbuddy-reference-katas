# Change Maker — TypeScript Walkthrough

This is an **algorithmic kata**: the input is a `number` amount and a `readonly number[]` of denominations, the output is a `number[]` of coins — so there are no domain builders, no value types, and no collaborators to introduce. The inputs and outputs *are* the domain. The reference lands as a single commit: `src/changeMaker.ts` exports `makeChange(amount, denominations)` that walks denominations largest-to-smallest, taking each coin while it fits in the remaining amount. `tests/changeMaker.test.ts` has one `it()` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); three `as const` coin tuples (`usCoins`, `ukCoins`, `norwayCoins`) keep test names in business vocabulary rather than anonymous numeric literals.

**Greedy is correct here, not generally.** The algorithm is a textbook greedy: largest coin that fits, repeat. For the three canonical currencies in this kata greedy yields the minimum-coin answer. For a non-canonical set like `[1, 3, 4]` making `6`, greedy picks `[4, 1, 1]` while the optimum is `[3, 3]` — solving the general minimum-coin problem requires dynamic programming. This reference deliberately stops at the canonical-currencies scope of the TDD Buddy prompt.

**Inline literals — deliberate.** The coin arrays are kept as module-local `as const` tuples rather than extracted into a `currencies` module. In F1 katas the test fixture fits on one screen and the coin lists *are* the test data — wrapping them would hide the vocabulary the test names already carry. F3 and larger modes prefer typed value objects; F1 deliberately doesn't.
