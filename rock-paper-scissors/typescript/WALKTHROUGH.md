# Rock Paper Scissors — TypeScript Walkthrough

This is an **algorithmic kata** with a small closed domain: two string-literal unions — `type Play = 'rock' | 'paper' | 'scissors'` and `type Outcome = 'win' | 'lose' | 'draw'` — and a pure function `decide(p1: Play, p2: Play): Outcome` that returns the result from player 1's perspective. There are no builders, no value types beyond the two type aliases, and no collaborators — the unions *are* the domain. The reference lands as a single commit: `src/rockPaperScissors.ts` declares both unions and `decide`, which short-circuits on equal plays as `'draw'`, then consults a `Record<Play, Play>` beats table — `rock → scissors`, `scissors → paper`, `paper → rock` — returning `'win'` if player 1's play beats player 2's and `'lose'` otherwise. `tests/rockPaperScissors.test.ts` has one `it()` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); all nine cells of the 3×3 matrix are enumerated as discrete tests for sibling parity with `fizz-buzz-whiz`.

**Typed unions over strings — deliberate.** F1 normally keeps inputs as primitives, but here the domain has a closed set of three plays and three outcomes. String-literal unions give TypeScript exhaustiveness without introducing runtime classes; the union names *are* the ubiquitous language, and the compiler rejects `decide('Rock', 'paper')` at the call site.
