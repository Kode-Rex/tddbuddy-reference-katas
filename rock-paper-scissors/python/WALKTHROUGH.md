# Rock Paper Scissors — Python Walkthrough

This is an **algorithmic kata** with a small closed domain: two `StrEnum`s — `Play(ROCK, PAPER, SCISSORS)` and `Outcome(WIN, LOSE, DRAW)` — and a pure function `decide(p1: Play, p2: Play) -> Outcome` that returns the result from player 1's perspective. There are no builders, no value types beyond the two enums, and no collaborators — the enums *are* the domain. The reference lands as a single commit: `src/rock_paper_scissors/rock_paper_scissors.py` defines both enums, a module-level `_BEATS` dict (`rock → scissors`, `scissors → paper`, `paper → rock`), and `decide`, which short-circuits on equal plays as `Outcome.DRAW`, then consults the beats table for `WIN` vs `LOSE`. The package `__init__.py` re-exports the three names so tests import them as `from rock_paper_scissors import Play, Outcome, decide`. `tests/test_rock_paper_scissors.py` has one function per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); all nine cells of the 3×3 matrix are enumerated as discrete tests for sibling parity with `fizz-buzz-whiz`.

**`StrEnum` over strings — deliberate.** F1 normally keeps inputs as primitives, but here the domain has a closed set of three plays and three outcomes with exact names the business uses. `StrEnum` gives a closed type with string-valued members — readable in repr, usable with `is` against the enum members, and callers can't accidentally pass `"Rock"` or `"ROCK"`. The enum names *are* the ubiquitous language.
