# IP Validator — Python Walkthrough

This is an **algorithmic kata**: the input is a `str` and the output is a `bool`, so there are no domain builders, no value types, and no collaborators to introduce — the inputs and outputs *are* the domain. The reference lands as a single commit: `src/ip_validator/ip_validator.py` defines `is_valid(input: str) -> bool`. The function splits on `.`, rejects anything that isn't exactly four segments, and then walks each octet through a private `_parse_octet` that returns `None` on empty octets, leading zeros, non-digits, or values above 255 — reusing the overflow check as the >255 guard so there's a single running total per octet. A final check rejects a last octet of `0` or `255` (network and broadcast). The package `__init__.py` re-exports `is_valid` so tests import it as `from ip_validator import is_valid`. `tests/test_ip_validator.py` has one function per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); each test name reads as a sentence from that spec.

**No regex — deliberate.** The TDD Buddy prompt forbids it, and hand-rolling the digit scan makes the leading-zero and negative-sign rejections fall out naturally from the character loop (a `-` is simply not a digit). `int(octet)` alone would accept `'007'`, `'+1'`, and `' 1 '`, and `str.isdigit()` accepts Unicode digits like `'²'` that aren't ASCII `0`–`9`. **Inline literals — deliberate.** The bounds `0`, `255`, and the octet count `4` are kept inline rather than extracted as named constants. In F1 katas the function body fits on one screen and the literals *are* the rule.
