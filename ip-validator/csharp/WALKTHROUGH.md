# IP Validator — C# Walkthrough

This is an **algorithmic kata**: the input is a `string` and the output is a `bool`, so there are no domain builders, no value types, and no collaborators to introduce — the inputs and outputs *are* the domain. The reference lands as a single commit: `src/IpValidator/IpValidator.cs` exposes `public static bool IsValid(string input)` on a static `IpValidator` class. The method splits on `.`, rejects anything that isn't exactly four segments, and then walks each octet through a local `TryParseOctet` that rejects empties, leading zeros, non-digits, and values above 255 — reusing the overflow check as the >255 guard so there's a single running total per octet. A final check rejects a last octet of `0` or `255` (network and broadcast). `tests/IpValidator.Tests/IpValidatorTests.cs` has one `[Fact]` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); each test name reads as a sentence from that spec.

**No regex — deliberate.** The TDD Buddy prompt forbids it, and hand-rolling the digit scan makes the leading-zero and negative-sign rejections fall out naturally from the character loop (a `-` is simply not a digit). `int.TryParse` alone would accept `"007"` and `" 1 "`, neither of which are valid here, so a bespoke parse is actually clearer than relying on the standard library. **Inline literals — deliberate.** The bounds `0`, `255`, and the octet count `4` are kept inline rather than extracted as named constants. In F1 katas the function body fits on one screen and the literals *are* the rule — naming `MaxOctet = 255` would write "255" twice without adding meaning.
