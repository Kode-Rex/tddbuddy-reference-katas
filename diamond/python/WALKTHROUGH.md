# Diamond — Python Walkthrough

This is an **algorithmic kata**: the input is a single-character string and the output is a multi-line ASCII string, so there are no domain builders, no value types, and no collaborators to introduce — the inputs and outputs *are* the domain. The reference lands as a single commit: `src/diamond/diamond.py` defines `print_diamond(letter)` — named to avoid shadowing the built-in `print` — which rejects non-single-character strings and anything outside `A`–`Z` / `a`–`z` with `ValueError` (message `letter must be a single A-Z character` — byte-identical across languages; exception type is idiomatic per language), normalizes with `.upper()`, then walks rows `0..2n` computing each row's offset as `min(r, 2n - r)` and building the row from string-multiplication primitives: `" " * (n - offset)` leading spaces, the row letter, then — when `offset > 0` — `" " * (2 * offset - 1)` inner spaces and a second copy of the letter. No trailing whitespace; lines joined with `"\n"` and no terminating newline. The package `__init__.py` re-exports `print_diamond` so tests import `from diamond import print_diamond`. `tests/test_diamond.py` has one function per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); each test name reads as a sentence from that spec.

**Inline literals — deliberate.** The mirror-offset arithmetic (`2 * offset - 1`, `2 * n + 1`) is itself the rule — naming `inner_space_count = 2 * offset - 1` just restates what the code already says. F3 and larger modes prefer named constants; F1 deliberately doesn't.
