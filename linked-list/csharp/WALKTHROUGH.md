# Linked List — C# Walkthrough

This is an **algorithmic kata** that bends the F1 shape slightly: the SUT is a data structure with identity and mutable state, not a pure function. The discipline that carries through is "no builders, no auxiliary domain types" — tests construct scenarios by calling the list's own operations (`Append`, `Prepend`, `Remove`, `InsertAt`), so the list **is** the SUT. The reference lands as a single commit. `src/LinkedList/LinkedList.cs` exposes `LinkedList<T>` with the methods named in [`../SCENARIOS.md`](../SCENARIOS.md). A private nested `Node` sealed class holds `Value` and `Next`; it is an implementation detail of the list and is never exposed — pushing `Node` to a top-level type would leak storage into the API for no test benefit. Out-of-range errors throw `ArgumentOutOfRangeException` with the byte-identical message `index out of range: {index}` that the other languages use. A small `ListOf(params int[])` helper in the test file reduces arrangement boilerplate without crossing into builder territory — it is only `Append` in a loop.

**Inline literals — deliberate.** Scenario literals like `1, 2, 3` and `[5, 7, 2]` sit inline in the tests rather than behind named constants. In F1 katas the inputs *are* the domain — naming `Triple = new[] { 1, 2, 3 }` writes the sequence twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
