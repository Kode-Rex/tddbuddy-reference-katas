# Linked List — Python Walkthrough

This is an **algorithmic kata** that bends the F1 shape slightly: the SUT is a data structure with identity and mutable state, not a pure function. The discipline that carries through is "no builders, no auxiliary domain types" — tests construct scenarios by calling the list's own operations (`append`, `prepend`, `remove`, `insert_at`), so the list **is** the SUT. The reference lands as a single commit. `src/linked_list/linked_list.py` defines `LinkedList` as a `Generic[T]` class with the methods named in [`../SCENARIOS.md`](../SCENARIOS.md). A module-private `_Node` class (also generic, with `__slots__`) holds `value` and `next`; the leading underscore keeps the storage shape out of the package's public surface. The package `__init__.py` re-exports `LinkedList` so tests import it as `from linked_list import LinkedList`. Out-of-range errors raise `IndexError` with the byte-identical message `index out of range: {index}` that the other languages use. Method names follow PEP 8 snake_case: `insert_at`, `index_of`, `to_array` — the C# and TypeScript implementations use `InsertAt` / `insertAt`, `IndexOf` / `indexOf`, `ToArray` / `toArray` respectively; the operation names in `SCENARIOS.md` are the abstract spec and each language renders them idiomatically. A small `list_of(*values)` helper in the test file reduces arrangement boilerplate without crossing into builder territory — it is only `append` in a loop.

**Inline literals — deliberate.** Scenario literals like `[1, 2, 3]` and `[5, 7, 2]` sit inline in the tests rather than behind named constants. In F1 katas the inputs *are* the domain — naming `TRIPLE = [1, 2, 3]` writes the sequence twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
