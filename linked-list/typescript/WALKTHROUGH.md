# Linked List — TypeScript Walkthrough

This is an **algorithmic kata** that bends the F1 shape slightly: the SUT is a data structure with identity and mutable state, not a pure function. The discipline that carries through is "no builders, no auxiliary domain types" — tests construct scenarios by calling the list's own operations (`append`, `prepend`, `remove`, `insertAt`), so the list **is** the SUT. The reference lands as a single commit. `src/linkedList.ts` exports `LinkedList<T>` with the methods named in [`../SCENARIOS.md`](../SCENARIOS.md). A module-private `Node<T>` class holds `value` and `next`; it is kept unexported so the storage shape never leaks into the API. Out-of-range errors throw `RangeError` with the byte-identical message `index out of range: {index}` that the other languages use. A small `listOf(...values)` helper in the test file reduces arrangement boilerplate without crossing into builder territory — it is only `append` in a loop.

**Inline literals — deliberate.** Scenario literals like `[1, 2, 3]` and `[5, 7, 2]` sit inline in the tests rather than behind named constants. In F1 katas the inputs *are* the domain — naming `const TRIPLE = [1, 2, 3]` writes the sequence twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
