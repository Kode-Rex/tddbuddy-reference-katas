# Character Copy — C# Walkthrough

This is the one **F1 kata with collaborators**. The SUT is not a pure function — it is a procedure that coordinates two small interfaces — so the reference introduces `ISource` and `IDestination` in `src/`, each with a single method, and ships hand-rolled fakes (`StringSource`, `RecordingDestination`) in `tests/`. The copier itself is a three-line loop in `src/CharacterCopy/CharacterCopy.cs`: read a char, stop on `'\n'`, otherwise write and continue. Six `[Fact]` tests in `tests/CharacterCopy.Tests/CharacterCopyTests.cs` map 1:1 to [`../SCENARIOS.md`](../SCENARIOS.md); each constructs a `StringSource` with the literal payload, runs `CharacterCopy.Copy`, and asserts on `destination.Written` (or `source.ReadCount` for the no-read-past-terminator scenario).

**Hand-rolled fakes — deliberate.** The original TDD Buddy prompt asks for a mocking framework; the reference implementation does not use one. A two-method interface with a five-line fake is clearer than any framework's `mock.Setup().Returns(...)` chain, and assertions on `destination.Written` read as a sentence about the domain (*"the destination wrote 'abc'"*) rather than about call sequences. This is the F1 take on the "mocking" kata: the collaborator interfaces are the teaching point, not the framework. F3 katas with richer collaborators (`IClock`, `INotifier`) use the same hand-rolled-fake discipline at larger scale — see `bank-account/`.
