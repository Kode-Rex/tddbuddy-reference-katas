# Character Copy — Python Walkthrough

This is the one **F1 kata with collaborators**. The SUT is not a pure function — it is a procedure that coordinates two small `Protocol` classes — so the reference introduces `Source` and `Destination` in `src/character_copy/`, each with a single method (`read() -> str`, `write(ch: str) -> None`), and ships hand-rolled fakes (`StringSource`, `RecordingDestination`) in `tests/`. The copier itself is a three-line loop in `src/character_copy/character_copy.py`: read a char, stop on `"\n"`, otherwise write and continue. The package `__init__.py` re-exports `copy`, `Source`, and `Destination` so tests import `from character_copy import copy`. Six pytest functions in `tests/test_character_copy.py` map 1:1 to [`../SCENARIOS.md`](../SCENARIOS.md); each constructs a `StringSource` with the literal payload, runs `copy`, and asserts on `destination.written` (or `source.read_count` for the no-read-past-terminator scenario). Python has no `char` primitive — a single-character `str` is the idiomatic choice, matching the TypeScript shape.

**Hand-rolled fakes — deliberate.** The original TDD Buddy prompt asks for a mocking framework; the reference implementation does not use one. A two-method `Protocol` with a five-line fake is clearer than any `MagicMock.side_effect = [...]` chain, and assertions on `destination.written` read as a sentence about the domain (*"the destination wrote 'abc'"*) rather than about call sequences. This is the F1 take on the "mocking" kata: the collaborator protocols are the teaching point, not the framework. F3 katas with richer collaborators (`Clock`, `Notifier`) use the same hand-rolled-fake discipline at larger scale — see `bank-account/`.
