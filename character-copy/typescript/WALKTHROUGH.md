# Character Copy — TypeScript Walkthrough

This is the one **F1 kata with collaborators**. The SUT is not a pure function — it is a procedure that coordinates two small interfaces — so the reference introduces `Source` and `Destination` in `src/`, each with a single method (`read(): string`, `write(ch: string): void`), and ships hand-rolled fakes (`StringSource`, `RecordingDestination`) in `tests/`. The copier itself is a three-line loop in `src/characterCopy.ts`: read a char, stop on `'\n'`, otherwise write and continue. Six `it()` tests in `tests/characterCopy.test.ts` map 1:1 to [`../SCENARIOS.md`](../SCENARIOS.md); each constructs a `StringSource` with the literal payload, runs `copy`, and asserts on `destination.written` (or `source.readCount` for the no-read-past-terminator scenario). TypeScript has no `char` primitive — a single-character `string` is the idiomatic choice, and the fakes enforce that discipline by returning one character at a time.

**Hand-rolled fakes — deliberate.** The original TDD Buddy prompt asks for a mocking framework; the reference implementation does not use one. A two-method interface with a five-line fake is clearer than any `vi.fn().mockReturnValueOnce(...)` chain, and assertions on `destination.written` read as a sentence about the domain (*"the destination wrote 'abc'"*) rather than about call sequences. This is the F1 take on the "mocking" kata: the collaborator interfaces are the teaching point, not the framework. F3 katas with richer collaborators (`Clock`, `Notifier`) use the same hand-rolled-fake discipline at larger scale — see `bank-account/`.
