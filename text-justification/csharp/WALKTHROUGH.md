# Text Justification — C# Walkthrough

This is an **algorithmic string-transform kata**: the input is a string plus an integer width, the output is a list of strings, and there are no aggregates, value types, or collaborators — the inputs and outputs *are* the domain. The reference lands as a single commit: `src/TextJustification/TextJustifier.cs` exposes `TextJustifier.Justify(string text, int width)` returning `IReadOnlyList<string>`. A greedy line-packer accumulates words until the next word would push the line past `width`, then calls `JustifyLine` to distribute the padding. The last line is emitted via `string.Join(' ', lineWords)` — left-aligned, single-spaced, not right-padded. `tests/TextJustification.Tests/TextJustifierTests.cs` has one `[Fact]` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md); each test name reads as a sentence from that spec.

**Padding distribution — integer division, left-heavy remainder.** The spec pads gaps evenly with the remainder landing on the left. `baseSpaces = padding / gaps; extras = padding % gaps;` then the first `extras` gaps each take one additional space. This is the standard LeetCode-68 shape; naming the two quantities makes the rule legible without extracting a helper type. Right-heavy distribution (extras on the last gaps) would read as a mirror of the same rule — the spec picks left-heavy because that matches the "fill from the left" intuition a typesetter reads the paragraph in.

**Tokenization via `Split((char[]?)null, StringSplitOptions.RemoveEmptyEntries)`.** Passing `null` for the separator array is the .NET idiom for "split on any Unicode whitespace"; `RemoveEmptyEntries` drops the empty tokens produced by runs of consecutive whitespace. This handles scenarios 1, 2, and 7 (empty, whitespace-only, and multi-space inputs) with a single API call and no special-casing.

**Single-word lines and oversize words.** `JustifyLine` branches once on `lineWords.Count == 1`: if the word is shorter than `width` it pads right with `PadRight(width)`; if it meets or exceeds `width` it is emitted unmodified (an oversize word is allowed to overflow rather than being split mid-word). The branch is expressed inline because it is the rule — extracting `PadSingleWord` would name a one-line conditional without adding meaning.

**Inline literals — deliberate.** The whitespace set is delegated to the framework via `null` separators, and the space character used for padding is `' '` inline in `StringBuilder.Append(' ', spaces)` and `string.Join(' ', ...)`. Naming `SPACE = ' '` would substitute a constant name for a character that *is* the rule. F1 katas treat the literal rules as the rule itself.
