# URL Parts — Python Walkthrough

This is an **algorithmic kata** in F1 mode, but with a twist: the output is not a primitive. A URL decomposes into seven fields, so `parse` returns a `UrlParts` dataclass — not a builder, just a data shape. Tests compare whole objects with `==`, which turns the scenario table in [`../SCENARIOS.md`](../SCENARIOS.md) into ten one-liner assertions. `src/url_parts/url_parts.py` defines `UrlParts` as a `@dataclass(frozen=True)` alongside `parse(url: str) -> UrlParts`, and `__init__.py` re-exports both. The parse is a left-to-right strip: `index('://')` splits protocol from rest, then `find('#')` peels off the anchor, `find('?')` the parameters, `find('/')` the path, `find(':')` the port, and finally the host resolves into subdomain + domain by matching one of the seven recognized TLDs or the bare literal `localhost`. Each split is a single `find` + slice pair — the parse rules stay visible without a state machine or recursive-descent scaffold.

**No regex, no `urllib.parse` — deliberate.** The TDD Buddy prompt forbids the built-in, and regex would hide the left-to-right peel behind one opaque pattern. A hand-rolled parse lets the reader see the order of operations (anchor, then query, then path, then port, then host) which is itself the algorithm. **TLD tuple and port dict are named constants — deliberate.** Unlike most F1 katas where inline literals are the rule, the seven TLDs and four default ports *are* domain knowledge ("these are the protocols and TLDs we recognize"), so named `TOP_LEVEL_DOMAINS` and `DEFAULT_PORTS` earn their keep. `int(...)` is fine for the port because we control the input (already-stripped digit tail of the authority).
