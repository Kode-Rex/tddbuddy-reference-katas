# URL Parts — TypeScript Walkthrough

This is an **algorithmic kata** in F1 mode, but with a twist: the output is not a primitive. A URL decomposes into seven fields, so `parse` returns a typed `UrlParts` interface — not a builder, just a data shape. Tests read the object directly with `toEqual({...})`, which turns the scenario table in [`../SCENARIOS.md`](../SCENARIOS.md) into ten one-liner assertions. `src/urlParts.ts` exports `parse(url: string): UrlParts` and the `UrlParts` interface from the same file so everything lives on one screen. The parse is a left-to-right strip: find `://` to split protocol from rest, `indexOf('#')` to peel off the anchor, `indexOf('?')` for parameters, `indexOf('/')` for the path, `indexOf(':')` for the port, then resolve the host into subdomain + domain by matching one of the seven recognized TLDs or the bare literal `localhost`. Each split is a single `indexOf` + `slice` pair — the parse rules stay visible without a state machine or recursive-descent scaffold.

**No regex, no `new URL()` — deliberate.** The TDD Buddy prompt forbids the built-in class, and regex would hide the left-to-right peel behind a single opaque pattern. A hand-rolled parse lets the reader see the order of operations (anchor, then query, then path, then port, then host) which is itself the algorithm. **TLD list and port table are named constants — deliberate.** Unlike most F1 katas where inline literals are the rule, the seven TLDs and four default ports *are* domain knowledge ("these are the protocols and TLDs we recognize"), so named `TOP_LEVEL_DOMAINS` and `DEFAULT_PORTS` earn their keep. `Number(...)` is fine for the port because we control the input (already-stripped digit tail of the authority).
