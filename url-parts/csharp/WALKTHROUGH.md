# URL Parts — C# Walkthrough

This is an **algorithmic kata** in F1 mode, but with a twist: the output is not a primitive. A URL decomposes into seven fields, so `Parse` returns a typed `UrlParts` record — not a builder, just a data shape. Tests read the record directly (`parts.Should().Be(new UrlParts(...))`), which turns the scenario table in [`../SCENARIOS.md`](../SCENARIOS.md) into ten one-liner assertions. `src/UrlParts/UrlParts.cs` exposes `public static UrlParts Parse(string url)` on a static `UrlParser` class, with the record declared in the same file so everything lives on one screen. The parse is a left-to-right strip: find `://` to split protocol from rest, split `#` to peel off the anchor, split `?` to peel off parameters, split `/` to peel off the path, split `:` to peel off the port, then resolve the host into subdomain + domain by matching one of the seven recognized TLDs or the bare literal `localhost`. Each split is a single `IndexOf` + slice — the parse rules stay visible without a state machine or recursive-descent scaffold.

**No regex, no `System.Uri` — deliberate.** The TDD Buddy prompt forbids the built-in class, and regex would hide the left-to-right peel behind a single opaque pattern. A hand-rolled parse lets the reader see the order of operations (anchor, then query, then path, then port, then host) which is itself the algorithm. **TLD list is a named constant — deliberate.** Unlike most F1 katas where inline literals are the rule, the seven TLDs *are* domain knowledge ("these are the TLDs we recognize"), so a named `TopLevelDomains` array earns its keep. The port defaults stay inline in a `switch` expression because the mapping is so small that the `switch` table is clearer than a dictionary.
