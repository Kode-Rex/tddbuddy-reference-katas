# Numbers to Words — C# Walkthrough

This is an **algorithmic kata**: the input is an `int` in the range `0..9999`, the output is the English-words spelling — so there are no domain builders, no value types, and no collaborators to introduce. The inputs and outputs *are* the domain. The reference lands as a single commit: `src/NumbersToWords/NumbersToWords.cs` exposes a static `ToWords(int n)` that splits the number into thousands, hundreds, and the sub-hundred remainder, appending each non-empty part to a list before joining with a single space. Two lookup arrays carry the whole lexicon — `Ones[0..19]` folds the teens into the same table as the single digits (so `nineteen` is one word, not `ten-nine`), and `Tens[2..9]` covers `twenty`..`ninety`. The hyphen rule drops out of one line: `ones == 0 ? Tens[tens] : $"{Tens[tens]}-{Ones[ones]}"`. `tests/NumbersToWords.Tests/NumbersToWordsTests.cs` has one `[Fact]` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md), grouped by digit count so a reader walks the grammar rules from left to right.

**Inline literals — deliberate.** The two word tables are private `static readonly string[]` fields on the class rather than being extracted into a `Lexicon` type. In F1 katas the implementation fits on one screen and the tables *are* the rule — wrapping them in a helper type would hide the business vocabulary the function body already carries. F3 and larger modes prefer typed value objects; F1 deliberately doesn't. The `"and"`-free hundreds join (`"three hundred three"`, not `"three hundred and three"`) follows the examples in the TDD Buddy kata prompt; American English style guides split on this, and the scenarios pin the choice explicitly.
