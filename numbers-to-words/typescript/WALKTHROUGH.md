# Numbers to Words — TypeScript Walkthrough

This is an **algorithmic kata**: the input is a `number` in the range `0..9999`, the output is the English-words spelling — so there are no domain builders, no value types, and no collaborators to introduce. The inputs and outputs *are* the domain. The reference lands as a single commit: `src/numbersToWords.ts` exports `toWords(n)` which splits the number into thousands, hundreds, and the sub-hundred remainder, pushing each non-empty part onto a list before joining with a single space. Two `as const` lookup tables carry the whole lexicon — `ONES[0..19]` folds the teens into the same array as the single digits (so `nineteen` is one word, not `ten-nine`), and `TENS[2..9]` covers `twenty`..`ninety`. The hyphen rule drops out of one expression: `ones === 0 ? TENS[tens] : \`${TENS[tens]}-${ONES[ones]}\``. With `noUncheckedIndexedAccess` on, the indexed reads are proven safe by the preceding bounds checks and asserted with `!`. `tests/numbersToWords.test.ts` has one `it()` per scenario in [`../SCENARIOS.md`](../SCENARIOS.md), grouped by digit count so a reader walks the grammar rules from left to right.

**Inline literals — deliberate.** The two word tables are module-local `as const` tuples rather than exports from a `lexicon` module. In F1 katas the implementation fits on one screen and the tables *are* the rule — wrapping them in a helper module would hide the business vocabulary the function body already carries. F3 and larger modes prefer typed value objects; F1 deliberately doesn't. The `"and"`-free hundreds join (`"three hundred three"`, not `"three hundred and three"`) follows the examples in the TDD Buddy kata prompt; American English style guides split on this, and the scenarios pin the choice explicitly.
