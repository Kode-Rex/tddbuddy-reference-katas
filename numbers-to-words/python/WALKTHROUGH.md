# Numbers to Words — Python Walkthrough

This is an **algorithmic kata**: the input is an `int` in the range `0..9999`, the output is the English-words spelling — so there are no domain builders, no value types, and no collaborators to introduce. The inputs and outputs *are* the domain. The reference lands as a single commit: `src/numbers_to_words/numbers_to_words.py` defines `to_words(n)` which splits the number with two `divmod` calls into thousands, hundreds, and the sub-hundred remainder, appending each non-empty part to a list before `" ".join`-ing. Two module-level tuples carry the whole lexicon — `_ONES[0..19]` folds the teens into the same tuple as the single digits (so `nineteen` is one word, not `ten-nine`), and `_TENS[2..9]` covers `twenty`..`ninety`. The hyphen rule drops out of one expression: `_TENS[tens] if ones == 0 else f"{_TENS[tens]}-{_ONES[ones]}"`. The package `__init__.py` re-exports `to_words` so tests import it as `from numbers_to_words import to_words`. `tests/test_numbers_to_words.py` has one function per scenario in [`../SCENARIOS.md`](../SCENARIOS.md), grouped by digit count so a reader walks the grammar rules from left to right.

**Inline literals — deliberate.** The two word tuples are module-level constants rather than being pulled into a `lexicon` module. In F1 katas the implementation fits on one screen and the tuples *are* the rule — wrapping them would hide the business vocabulary the function body already carries. F3 and larger modes prefer typed value objects; F1 deliberately doesn't. The `"and"`-free hundreds join (`"three hundred three"`, not `"three hundred and three"`) follows the examples in the TDD Buddy kata prompt; American English style guides split on this, and the scenarios pin the choice explicitly.
