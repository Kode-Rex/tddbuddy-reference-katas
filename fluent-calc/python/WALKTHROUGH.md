# Fluent Calc — Python Walkthrough

This is an **algorithmic kata** that bends the F1 shape slightly: the SUT is a stateful fluent API, not a pure function. The discipline that carries through is "no builders, no auxiliary domain types" — **the chain itself is the builder**. Tests assemble scenarios by calling `Calculator`'s own methods (`seed`, `plus`, `minus`, `undo`, `redo`, `save`), so there is no test-side arrangement helper worth writing. The reference lands as a single commit. `src/fluent_calc/calculator.py` defines `Calculator` with one chained method per verb in [`../SCENARIOS.md`](../SCENARIOS.md); every chained method returns `self` — the Pythonic shape for a mutable fluent object. Undo is modeled as a pair of `list[_Operation]` stacks (`.append`/`.pop`) — an undo stack of applied operations and a redo stack of reverted ones — with a module-private `_Operation` frozen dataclass (`kind: Literal["plus", "minus"]`, `operand: int`) that module-level `_forward`/`_reverse` helpers interpret. A new `plus` or `minus` appends to the undo stack and clears the redo stack, matching the classic undo/redo invariant. `save` clears both stacks with `.clear()` rather than layering a "saved-before" sentinel, which keeps `undo`/`redo` branch-free — they always operate on the live stacks. **The calculator never throws**: unseeded `plus`/`minus` and empty-stack `undo`/`redo` return `self` untouched, matching the kata's "never throws" rule. The package `__init__.py` re-exports `Calculator` so tests import it as `from fluent_calc import Calculator`.

**Progressive interfaces considered, rejected.** The TDD Buddy hint suggests returning a narrower protocol after `seed` to gate repeat-`seed` at type-check time. That pattern earns its keep in larger builder-driven domains where the type system is doing real scenario-validation work; at F1's one-screen scope (and in Python, where runtime duck-typing is the norm) the structural-typing machinery would double the file size to save one test. The runtime ignore is tested and matches the spec's "ignored" wording verbatim.

**Inline literals — deliberate.** Scenario values like `.seed(10).plus(5).minus(2)` sit inline in the tests rather than behind named constants. In F1 katas the inputs *are* the domain — naming `BASE = 10` writes "ten" twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
