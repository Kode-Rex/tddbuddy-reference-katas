# Fluent Calc — TypeScript Walkthrough

This is an **algorithmic kata** that bends the F1 shape slightly: the SUT is a stateful fluent API, not a pure function. The discipline that carries through is "no builders, no auxiliary domain types" — **the chain itself is the builder**. Tests assemble scenarios by calling `Calculator`'s own methods (`seed`, `plus`, `minus`, `undo`, `redo`, `save`), so there is no test-side arrangement helper worth writing. The reference lands as a single commit. `src/calculator.ts` exports `Calculator` with one chained method per verb in [`../SCENARIOS.md`](../SCENARIOS.md); every chained method returns `this` with `this`-type so the chain preserves the concrete class and a subclass's additions would remain visible. Undo is modeled as a pair of `Operation[]` arrays used as stacks (`.push`/`.pop`) — an undo stack of applied operations and a redo stack of reverted ones — with a module-local `Operation` interface that a pair of `forward`/`reverse` functions interprets. A new `plus` or `minus` pushes onto undo and clears redo, matching the classic undo/redo invariant. `save` clears both stacks by assigning `.length = 0` rather than layering a "saved-before" sentinel, which keeps `undo`/`redo` branch-free — they always operate on the live stacks. **The calculator never throws**: unseeded `plus`/`minus` and empty-stack `undo`/`redo` return `this` untouched, matching the kata's "never throws" rule.

**Progressive interfaces considered, rejected.** The TDD Buddy hint suggests returning a narrower interface after `seed` to make repeat-`seed` a compile error. That pattern earns its keep in larger builder-driven domains where the type system is doing real scenario-validation work; at F1's one-screen scope it doubles the file size to save one test. The runtime ignore is tested and matches the spec's "ignored" wording verbatim.

**Inline literals — deliberate.** Scenario values like `.seed(10).plus(5).minus(2)` sit inline in the tests rather than behind named constants. In F1 katas the inputs *are* the domain — naming `const BASE = 10` writes "ten" twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
