# Fluent Calc — C# Walkthrough

This is an **algorithmic kata** that bends the F1 shape slightly: the SUT is a stateful fluent API, not a pure function. The discipline that carries through is "no builders, no auxiliary domain types" — **the chain itself is the builder**. Tests assemble scenarios by calling `Calculator`'s own methods (`Seed`, `Plus`, `Minus`, `Undo`, `Redo`, `Save`), so there is no test-side arrangement helper worth writing. The reference lands as a single commit. `src/FluentCalc/Calculator.cs` exposes `Calculator` with one chained method per verb in [`../SCENARIOS.md`](../SCENARIOS.md); every chained method returns `this`, which is the natural shape for mutable fluent state (an immutable variant would allocate a new object and two new stacks per chain step for no teaching benefit). Undo is modeled as a pair of `Stack<Operation>` fields — an `_undo` of applied operations and a `_redo` of reverted ones — with a private nested `Operation(Op Kind, int Operand) readonly record struct` that a private `Forward`/`Reverse` pair interprets. A new `Plus` or `Minus` pushes onto `_undo` and clears `_redo`, matching the classic undo/redo invariant. `Save` clears both stacks rather than layering a "saved-before" sentinel, which keeps `Undo`/`Redo` branch-free — they always operate on the live stacks. **The calculator never throws**: unseeded `Plus`/`Minus` and empty-stack `Undo`/`Redo` return `this` untouched, matching the kata's "never throws" rule.

**Progressive interfaces considered, rejected.** The TDD Buddy hint suggests returning a narrower interface after `Seed` to make repeat-`Seed` a compile error. That pattern earns its keep in larger builder-driven domains where the type system is doing real scenario-validation work; at F1's one-screen scope it doubles the file size to save one test. The runtime ignore is tested and matches the spec's "ignored" wording verbatim.

**Inline literals — deliberate.** Scenario values like `Seed(10).Plus(5).Minus(2)` sit inline in the tests rather than behind named constants. In F1 katas the inputs *are* the domain — naming `BaseValue = 10` writes "ten" twice without adding meaning. F3 and larger modes prefer named constants and builders; F1 deliberately doesn't.
